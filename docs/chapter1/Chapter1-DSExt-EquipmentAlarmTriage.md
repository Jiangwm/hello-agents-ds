# 第一章 DSExt：设备告警分诊智能体

> 难度：入门 ｜ 对应主题：智能体构成、感知—决策—行动循环、Workflow 与 Agent

## 1. 场景定位

某离散制造车间每天产生温度、振动、电流等设备告警。值班工程师希望用自然语言询问“冲压机 A 刚才为什么报警”，由一个最小智能体读取告警片段、查询告警规则，并给出初步分诊建议。

本场景只做**辅助分析**：智能体可以汇总数据、引用规则和建议检查项，但不得直接控制设备，也不得把推测写成已确认故障。

## 2. 与原章内容的静态映射

| 原章知识或代码证据 | 工业扩展映射 |
| --- | --- |
| `FirstAgentTest.py` 中的 `get_weather()` | 替换为按设备和时间窗读取传感器摘要的工具 |
| `get_attraction()` 与 `available_tools` | 替换为告警规则查询工具及显式工具注册表 |
| Thought—Action—Observation 循环 | 形成“理解告警—取数—查规则—给结论”的最小闭环 |
| Workflow 与 Agent 的区别 | 固定阈值检查走 Workflow；信息不足时的动态取证交给 Agent |

难度保持在“第一个智能体”层级：仅两个只读工具、单个智能体、最多三轮行动，不引入数据库、向量检索或多智能体协作。

## 3. 数据与问题定义

准备一份小型脱敏 CSV，字段包括：

- `timestamp`：采样时间；
- `equipment_id`：设备编号；
- `temperature_c`、`vibration_mm_s`、`current_a`：关键测点；
- `alarm_code`：PLC 或监控系统产生的告警码；
- `status`：运行、待机或停机状态。

另准备一份告警规则表，包含告警码、阈值、可能原因、人工检查项和安全等级。典型问题为：

1. 指定设备在告警前后 10 分钟出现了哪些异常？
2. 哪条规则与当前数据最匹配？
3. 还缺少什么数据才能提高判断置信度？

## 4. 智能体设计

智能体具备两个概念工具：

- `query_sensor_window(equipment_id, start, end)`：返回时间窗内的最小值、最大值、均值及越限点；
- `lookup_alarm_rule(alarm_code)`：返回规则来源、阈值和人工检查清单。

运行流程：

1. 从用户问题中识别设备、告警码与时间范围；
2. 缺少关键参数时先追问，不猜测设备或时间；
3. 调用传感器工具获得事实；
4. 调用规则工具获得规则依据；
5. 输出“数据事实—规则匹配—建议动作—不确定性”。

## 5. 实践任务

- 基础：完成单一告警码的查询与规则解释。
- 核心：让智能体自动决定先取数据还是先查规则。
- 挑战：加入“信息不足”分支，并限制最大工具调用次数。

输出报告至少应包含设备、时间窗、异常测点、引用的规则编号、建议人工检查项和风险声明。

## 7. 验收标准

- 对给定样例能完成至少一次“感知—行动—观察—回答”闭环；
- 所有数值结论可回溯到输入数据，所有规则结论可回溯到规则表；
- 不触发任何写控制或设备控制动作；
- 数据不足时明确说明未知项，不编造故障原因。
