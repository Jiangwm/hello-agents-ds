# 第四章 DSExt：生产异常根因分析智能体

> 难度：基础进阶 ｜ 对应主题：ReAct、Plan-and-Solve、Reflection 与工具系统

## 1. 场景定位

某产线的产品一次合格率突然下降。质量工程师希望智能体关联工艺参数、设备告警、批次信息和质检缺陷，形成可验证的根因假设与排查计划。

本章不追求完整平台，而是让同一工业问题分别经过三种经典范式，理解它们在开放探索、任务分解和结果复核方面的差异。

## 2. 与原章内容的静态映射

| 原章知识或代码证据 | 工业扩展映射 |
| --- | --- |
| `tools.py` 的 `ToolExecutor` | 注册批次查询、趋势统计、异常检测和工艺规范查询工具 |
| `ReAct.py` 的循环与输出解析 | 边观察数据边选择下一项证据 |
| `Plan_and_solve.py` 的 `Planner`/`Executor` | 先拆分批次对比、参数筛查、假设验证和报告步骤 |
| `Reflection.py` 的执行—评审记忆 | 复核数据引用、因果措辞和遗漏变量，再改写报告 |

## 3. 数据资产与工具边界

使用四类脱敏表：

- 批次主表：产品、产线、班次、物料批次、开始和结束时间；
- 工艺时序表：温度、压力、速度、流量等参数；
- 质量结果表：缺陷类型、测量值、合格判定；
- 事件表：告警、换料、维护、换班和程序版本。

工具仅允许读取与计算：`get_batch_profile`、`compare_time_windows`、`detect_parameter_shift`、`lookup_process_spec`。任何工艺调整必须由工程师批准。

## 4. 三种范式的任务设计

### 4.1 ReAct：探索式取证

智能体从“哪个批次开始恶化”出发，逐步查询趋势、相邻事件和规范限值。每次行动都必须写明希望验证的假设，观察结果必须带数据范围。

### 4.2 Plan-and-Solve：结构化排查

规划器先生成可执行步骤：确定异常窗口、构造正常对照、筛查显著漂移参数、核查共因事件、形成候选根因。执行器逐步产出中间结果，失败步骤不得被静默跳过。

### 4.3 Reflection：报告复核

评审器检查：是否把相关性写成因果、是否遗漏基线、是否引用错误批次、建议是否越过安全边界。修改后的报告保留复核前后的差异。

## 5. 输出格式

最终根因分析报告包含：

1. 事件范围与影响；
2. 正常组和异常组定义；
3. 数据证据及查询条件；
4. 候选根因排序与反证；
5. 建议补充采集的数据；
6. 需人工审批的验证实验。

建议用“已确认事实 / 数据支持的假设 / 尚无证据”三级标签控制措辞。

## 6. 实践任务

- 基础：为四个只读工具定义清晰输入输出。
- 核心：分别运行三种范式并比较调用次数、完整性和可审计性。
- 挑战：构造一个混杂因素案例，观察 Reflection 能否发现错误归因。

## 8. 验收标准

- 每个结论都能定位到工具结果或明确标记为假设；
- 三种范式解决的是同一数据问题，便于横向比较；
- 至少包含一个失败工具调用和一个证据不足分支；
- 不自动修改 PLC、MES、配方或工艺参数。
