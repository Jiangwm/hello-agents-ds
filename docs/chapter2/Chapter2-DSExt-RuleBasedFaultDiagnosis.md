# 第二章 DSExt：规则驱动的泵组故障问诊助手

> 难度：入门 ｜ 对应主题：符号主义、专家系统、ELIZA 模式匹配及其局限

## 1. 场景定位

为化工装置循环泵设计一个透明、可解释的故障问诊助手。操作员输入“出口压力下降并伴随振动升高”等现象，系统通过关键词与规则匹配，追问必要信息，并返回候选故障和检查顺序。

该助手用于教学和班组预检，不替代联锁系统、设备工程师或正式检修规程。

## 2. 与原章内容的静态映射

| 原章知识或代码证据 | 工业扩展映射 |
| --- | --- |
| 专家系统与产生式规则 | 将设备知识写成“现象组合 → 候选原因 → 检查项” |
| `ELIZA.py` 的正则规则库 | 匹配压力、流量、振动、温度和异响等描述 |
| `respond()` 的顺序匹配 | 体现规则优先级和兜底响应 |
| ELIZA 的通配规则 | 当证据不足时追问，而不是给出确定诊断 |

难度与原章一致：不使用大模型，不训练分类器，重点理解规则系统的可解释性、覆盖盲区和维护成本。

## 3. 规则与输入设计

输入采用简短自然语言，并允许附带结构化测量值。规则库至少覆盖：

- 汽蚀：入口压力偏低、噪声或振动升高、流量波动；
- 轴承异常：轴承温度与高频振动同时升高；
- 过滤器堵塞：入口压差增大、流量持续下降；
- 密封泄漏：泄漏量或密封腔温度异常；
- 证据冲突：不同规则命中但关键条件不足。

每条规则记录 `rule_id`、触发模式、必要条件、候选原因、风险等级、追问模板和检查步骤。

## 4. 问诊流程

1. 规范化单位、设备别名和常见口语表达；
2. 按安全规则、强特征规则、弱特征规则、兜底规则的顺序匹配；
3. 对缺失的必要条件发起一次针对性追问；
4. 输出命中规则、匹配证据及未满足条件；
5. 若没有高置信规则，则建议转人工并保留原始输入。

示例输出应区分：

- **观察事实**：操作员输入或测点值；
- **规则推断**：由哪条规则得到的候选故障；
- **下一步检查**：可安全执行的人工核查；
- **限制**：尚未采集的数据和规则库覆盖范围。

## 5. 实践任务

- 基础：将 `ELIZA.py` 的规则与响应模板替换为泵组规则。
- 核心：增加单位和同义词归一化，例如“振得厉害”映射到振动异常。
- 挑战：当多条规则命中时，按必要条件满足率和风险等级排序。

建议用 20 条人工编写问句进行离线验证，统计正确命中、错误命中、兜底和追问比例。

## 7. 验收标准

- 相同输入始终获得可复现的规则匹配结果；
- 输出明确列出命中的 `rule_id`；
- 无匹配时进入追问或人工升级路径；
- 高风险现象不得被低风险建议覆盖；
- 能说明规则系统为何难以覆盖模糊、组合和新型故障。
